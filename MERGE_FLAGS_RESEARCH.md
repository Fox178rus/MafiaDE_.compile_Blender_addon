# Merge-by-flag — исследование флагов split для окружения города

## Проблема
Меши окружения (не машины) хранятся в VB разрезанными на 2 половины по одной из осей.
Одна половина смещена на +128 единиц. Флаги в байтах VB указывают какая вершина к какой половине принадлежит.

## Решение (РАБОЧЕЕ — проверено на File_387.ires.compiled)

### stride=24 и stride=28
- **byte[5] bit7** (старший бит Z high byte) = флаг split по Z
- `fz=1` → вершина смещена на **+128 по Z** → нужно сдвинуть на **-128**
- `fz=0` → вершина на месте, не трогать

```python
# stride=24
fz = (data[off+5] >> 7) & 1
if fz == 1:
    v[2] -= 128.0

# stride=28
fz = (data[off+5] >> 7) & 1
if fz == 1:
    v[2] -= 128.0
```

### stride=52 (машины)
- Отдельная логика, не менять — работает через auto_merge

## Пример: File_387.ires.compiled
- VB@9229 stride=24, 558 вершин: fz=0 (479 вершин) Z=0..91, fz=1 (79 вершин) Z=191..213
- VB@33599 stride=28, 634 вершин: fz=0 (554 вершин) Z=0..96, fz=1 (80 вершин) Z=149..217
- После -128 для fz=1: обе группы собираются в Z=0..96

## Другие фиксы в scan_ires.py (рабочие)
- `KNOWN_STRIDES` — добавлен stride=16
- `MIN_VB_VERTS` — снижен с 200 до 50
- Пост-обработка буферов — переклассификация IB→VB если size == (max_vi+1)*stride

## ВАЖНО
Не трогать логику stride=52 (машины) — там другая система флагов через auto_merge.
Не добавлять Y merge для stride=24 — у всех вершин fy=0, это не флаг split.
